## Unify YouTube/video source detail view with the PDF source detail layout (#575)

### Problem

Issue #575 (PR #572) shipped the YouTube/Vimeo embed player + transcript as a
bespoke layout in `SourceDetailView`: a fixed-height player iframe stacked on
top of the transcript in one scroll column. Meanwhile, PDF sources had a much
richer detail view — a segmented **Reader / PDF / Split** tab picker with an
outline pane, zoom, and find. Video sources should use that _same_ layout, with
the video player standing in for PDFKit.

### What changed

Generalized the PDF tab system so byteless media embeds (YouTube/Vimeo/Spotify/
SoundCloud/direct-remote) route through the same tabbed viewer, and deleted the
bespoke stacked player+transcript layout. **One file changed:**
`Sources/WikiFS/Sources/SourceDetailView.swift`.

- **`FileContentTab`** — renamed `.markdown` → `.reader` (the tab shows the
  rendered transcript _or_ processed markdown; "Reader" matches the PDF-side
  label), added `.video`. Kept `.pdf` and `.split`.
- **`availableTabs`** (new computed `[FileContentTab]`) — replaces the boolean
  `showTabs`. A video source gets `[.reader, .video]` plus `.split` only when a
  transcript exists; a PDF with extraction keeps `[.reader, .pdf, .split]`; an
  un-extracted PDF / native markdown / binary source gets `[]` (bare content,
  no picker). `showTabs` is now `!availableTabs.isEmpty`.
- **`contentArea`** — the `if isBytelessEmbedWithPlayer … embedContent(…)`
  branch is gone; video sources fall into `showTabs || isBytelessEmbedWithPlayer`
  → `tabbedContent` (same path as PDF+extraction). PDF-only / native-markdown /
  binary branches untouched.
- **`tabbedContent`** — `.reader` shows the transcript reader (or
  `embedEmptyReaderContent` when a byteless embed has no transcript yet);
  `.video` shows `videoPlayerContent`; `.pdf`/`.split` unchanged.
- **`videoPlayerContent`** (new) — wraps `MediaEmbedPlayerView` (unchanged,
  reusable) as a full-pane tab body, with a `ContentUnavailableView` fallback
  when the embed target can't resolve. Replaces the deleted
  `embedContent(target:)`.
- **`embedEmptyReaderContent`** (new) — the calm "No Transcript Available"
  placeholder that fills the Reader tab for a video with no captions.
- **`splitContent`** — right pane now picks the source's visual companion: the
  PDF for PDF sources, the video player for byteless embeds. Left pane identical.
- **`selectedTab`** default + file-id reset → `.reader`; the Edit button's
  "switch away from a no-editor tab" guard now also covers `.video`.

### Why this is the right generalization

The player lives where the PDFKit view lived and the transcript lives where
processed-markdown lived. Everything else (toolbar, outline, zoom, find,
extraction chip, provenance) stays identical because it already keyed off
`headVersion` / `isPDF`, not the content layout. The outline populates from the
transcript's headings for free (`contentAndOutline` keys off
`currentMarkdownContent`, which is the transcript markdown). `MediaEmbedPlayerView`
+ `MediaEmbedPlayerHTML` are untouched — kept as the reusable WKWebView player
component; only the wrapping layout changed.

### What does NOT change

- PDF source detail view behavior — identical (same tabs, same outline, same
  extraction chip).
- `MediaEmbedPlayerView` / `MediaEmbedPlayerHTML` / `ExternalEmbed` — untouched.
- `WikiReaderView` — the transcript renders through the same reader (wiki
  links, footnotes, mermaid, ghost-link coloring, zoom, find all work).
- Store, schema, ingest, File Provider — no changes.

### Verification

- `make version prompts && swift build` — clean.
- Fast test tier: **2573 tests / 218 suites passed**.
- Targeted: `MediaEmbedPlayerTests | ExternalEmbedTests | YouTubeEmbedWebViewTests`
  — 34/34 pass.

### Notes

- Follows `macos-design`: native segmented tab picker preserved,
  `ContentUnavailableView` for empty states, no web-artifact patterns.
- Diagnostics routed through `DebugLog` (no `print`) — no new logging needed in
  this change (the view logs nothing itself).
- Did not merge to `main`; this PR targets `main`.

Closes the layout portion of #575.
