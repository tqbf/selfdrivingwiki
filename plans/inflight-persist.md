# Plan: Incremental In-Flight Turn Persistence (Issue #826)

**Goal:** A partial assistant turn survives an app close / crash mid-generation.
Streamed assistant text is checkpointed to SQLite *during* generation â€” not only
at turn completion â€” so a hard kill no longer drops the in-flight row.

**Status:** Implementing. Every claim cites `file:line`.

---

## 1. Root cause â€” why the in-flight turn is lost today

Chat persistence is incremental at **turn boundaries**, keyed on an
append-only count cursor. Streaming breaks both halves of that model.

### 1a. Streaming mutates the last row *in place* â€” `events.count` is stable

`mergeOrAppend` coalesces each `.assistantTextDelta` into the in-progress
`.assistantText` row by **overwriting the same index**
(`AgentLauncher.swift:3446-3453`):

```swift
case .assistantTextDelta(let delta):
    if isStreamingAssistantRow, case .assistantText(let existing) = events.last {
        events[events.count - 1] = .assistantText(existing + delta)   // same index!
    } else {
        events.append(.assistantText(delta))                          // +1 here only
        ...
        isStreamingAssistantRow = true
    }
```

So after the first delta appends a row, every subsequent delta **grows that
row without incrementing `events.count`**.

### 1b. The flush cursor is count-based â€” it cannot see in-place growth

`flushTranscript` gates on `persistedEventCount < events.count`
(`AgentLauncher.swift:3393-3396`). Because deltas don't move the count, once
the cursor catches up, no mid-stream mutation can ever reopen the `<` window.

### 1c. `flushTranscript` only fires at boundaries

Turn completion (`AgentLauncher.swift:3199-3201`), session teardown (`finish()`
at `AgentLauncher.swift:3543`), and on `stopAgent`. A close mid-generation
drops the in-memory partial row.

### 1d. The store is append-only â€” there is no upsert path

`appendChatMessages` assigns `seq = MAX(seq)+1` and a fresh ULID `id` per row
(`GRDBWikiStore.swift:6036-6054`). There is no method to update a row in place.

**Conclusion:** incremental mid-turn persistence needs (1) a content-aware
dirty signal (not the count cursor), (2) a timer/hybrid trigger that fires
mid-stream, and (3) a store-side upsert keyed on a stable handle so a growing
row updates the same DB row instead of appending duplicates.

---

## 2. Design â€” the "draft-handle upsert" model

A streaming row is checkpointed as a **draft row**: inserted on first
checkpoint, *updated in place* on every subsequent checkpoint, and
**finalized** (is_draft -> 0) at turn completion. The stable identity is a
**draft handle** (a ULID generated by the launcher when the streaming row is
created), used as a SQLite upsert key (manual SELECT/INSERT/UPDATE inside
`mutate()`, per correction C5 â€” matching existing `appendChatMessages`
style).

**Why a handle, not a seq?** The store assigns `seq` on insert; the launcher
would need it back (breaking the void sink). A launcher-generated ULID handle
is the upsert key; the store never returns anything.

**Why not a new event type / "draft" AgentEvent case?** `event_json` must stay
a valid, decodable `.assistantText(String)` so history reload renders drafts
with zero special-casing. "Draftness" is row-level metadata (`is_draft`),
orthogonal to the event payload.

## 3. Schema change â€” migration v43 -> v44 (correction C1)

#830/#849 already landed v43 (`chats.acp_session_id`). **This plan owns v43
-> v44.** `currentSchemaVersion` -> 44 (`GRDBWikiStore.swift:61`).

```sql
ALTER TABLE chat_messages ADD COLUMN is_draft    INTEGER NOT NULL DEFAULT 0;
ALTER TABLE chat_messages ADD COLUMN draft_handle TEXT;
CREATE UNIQUE INDEX IF NOT EXISTS chat_messages_draft_handle
    ON chat_messages(draft_handle) WHERE draft_handle IS NOT NULL;
```

- `is_draft INTEGER NOT NULL DEFAULT 0` â€” DEFAULT 0 keeps every pre-existing
  row finalized with no backfill.
- `draft_handle TEXT` â€” nullable; NULL for every append-only row. The partial
  unique index allows many NULLs while enforcing one row per live handle.
- Added to both the migration ladder AND `createFreshSchema`'s `chat_messages`
  DDL so fresh DBs match.

## 4. Store layer â€” new upsert mutator

New public method on `GRDBWikiStore` (and the `WikiStore` protocol, since
`WikiStoreModel` holds `store: WikiStore`), beside `appendChatMessages`
(`GRDBWikiStore.swift:6014`):

```swift
public func checkpointStreamingMessage(
    chatID: PageID, handle: String, event: AgentEvent, isDraft: Bool
) throws
```

Manual SELECT/INSERT/UPDATE inside `mutate()` (C5 â€” no ON CONFLICT). Mirrors
`appendChatMessages` style: existence check, lookup by `draft_handle`, then
INSERT (is_draft=1, nextSeq=MAX(seq)+1) or UPDATE (event_json, text, is_draft).
Routes through `mutate()` + emits `ResourceChangeEvent(.chat, .updated)`.

**C2 (critical):** the checkpoint sink reports `-> Bool` (success), and the
launcher only advances `persistedEventCount` when the sink returns `true`. On
`false`, `streamingRowDirty` stays `true` so the next checkpoint retries.

**C6 (medium):** draft checkpoints do NOT bump `chats.updated_at` (only the
finalize does), so the chat list doesn't re-sort every 2s.

**C8 (medium):** `finalizeStaleDrafts(forChat:)` â€” when a chat is opened,
`UPDATE chat_messages SET is_draft=0 WHERE chat_id=? AND is_draft=1`. Kept
meaningful over time.

## 5. Launcher layer

### 5a. New state fields (near `persistedEventCount`, `AgentLauncher.swift:678`)

```swift
private var streamingRowDirty = false
private var streamingDraftHandle: String?
private var checkpointTimer: Task<Void, Never>?
```

### 5b. Set dirty + assign handle in `mergeOrAppend`

In the `.assistantTextDelta` first-delta sub-case (before
`isStreamingAssistantRow = true`): assign
`streamingDraftHandle = ULID.generate()`. In both sub-cases:
`streamingRowDirty = true`.

**C7:** `.thinkingDelta` is scoped OUT for v1 (assistant-only). Documented.

**C3 (critical):** when a new streaming row begins while
`streamingDraftHandle != nil`, finalize the old handle BEFORE assigning the
new one.

In the `case .assistantText` finalization branch: set `streamingRowDirty =
true` (authoritative final text differs from last checkpoint) but keep
`streamingDraftHandle` so the final update lands on the same row.

### 5c. Checkpoint trigger â€” debounced timer (~2s, rearmed per delta)

```swift
private func scheduleCheckpoint() {
    checkpointTimer?.cancel()
    checkpointTimer = Task { @MainActor [weak self] in
        try? await Task.sleep(for: .seconds(2))
        guard !Task.isCancelled, let self else { return }
        self.checkpointStreamingRow(finalize: false)
        if self.isGenerating { self.scheduleCheckpoint() }
    }
}
```

Called from `setGenerating(true)`; cancelled in `setGenerating(false)`,
`finish()`, `resetRunArtifacts()`.

### 5d. `checkpointStreamingRow(finalize:)` â€” the mid-turn flush

```swift
private func checkpointStreamingRow(finalize: Bool) {
    guard let chatID = activeChatID,
          let handle = streamingDraftHandle,
          let last = events.last,
          case .assistantText = last,
          streamingRowDirty || finalize else { return }
    let ok = streamingCheckpointSink?(chatID, handle, last, finalize ? false : true) ?? false
    if ok {
        streamingRowDirty = false
        persistedEventCount = max(persistedEventCount, events.count)
    }
    if finalize { streamingDraftHandle = nil }
}
```

Critical interaction with the count cursor: checkpointing advances
`persistedEventCount` to `events.count`. Because the streamed row grows in
place (same index), the turn-end `flushTranscript()` tail naturally excludes
the already-checkpointed row.

### 5e. Turn-end finalize (`AgentLauncher.swift:3199`)

In the `endsGeneration` block, before `flushTranscript()`:

```swift
self.checkpointStreamingRow(finalize: true)
self.checkpointTimer?.cancel(); self.checkpointTimer = nil
self.flushTranscript()
```

### 5f. `finish()` teardown (`AgentLauncher.swift:3532`)

Add a finalize before the existing `flushTranscript()` so a session that ends
without a clean `.messageStop` still commits the final streamed text.

### 5g. Reset sites

Reset the new fields wherever `persistedEventCount` is reset:
`resetRunArtifacts()` (`AgentLauncher.swift:3656`), `finish()`.

## 7. Store-emit invariant

The checkpoint routes through `mutate()` and emits
`ResourceChangeEvent(.chat, .updated)` â€” it is a real content mutation. Per
C6, draft checkpoints skip the `chats.updated_at` bump (only finalize bumps
it), so `updated_at`-sensitive subscribers don't re-sort every 2s.

## 8. Crash-survival hardening (the core of #826)

| Scenario | What runs | What survives |
|---|---|---|
| Clean app close mid-turn | `finish()` -> finalize + flushTranscript | Final text, is_draft=0 |
| Hard kill / power loss mid-turn | (no `finish()`) | Last periodic checkpoint â€” partial text, is_draft=1 |
| Kill within first checkpoint window | nothing | Mitigated by timer cadence |
